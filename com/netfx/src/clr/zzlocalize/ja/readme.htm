マイクロソフトへのフィードバックの返し方

Microsoft® Visual Studio.NET Beta 1
Microsoft® .NET Framework SDK Beta 1

マイクロソフトでは、Microsoft Visual Studio.NET、または Microsoft .NET Framework SDKのベータをご利用になった皆様から、ご意見や障害情報等のフィードバックを広く募集しております。このベータに対するフィードバックは、電子メールで以下の要領でお送りください。

【宛先】 kkvsbf@microsoft.com
【件名】 先頭に必ず VS: と半角で付記してください。それ以降の形式は自由です。
【本文】 以下の形式を参考にして記述してください。

１．ベータ製品名

下記のどちらかを選んで記入します
  • Visual Studio.NET Beta 1 (または VS.NET Beta1)
  • .NET Framework SDK Beta 1 (または .NET SDK Beta1。SDK 単体で利用の場合のみ)

２．MSDN 会員の方

MSDN 会員の方は、以下の項目を記入してください。

  • MSDN 会員番号
  • MSDN 会員登録者名

３．感想をお送りいただく場合

最初の一行は「感想」とのみ書いてください。初めてフィードバックをお送りいただく場合、わかる範囲でハードウェアの仕様 (CPU、メモリ容量、DVD ドライブの有無など) やお使いの OS の種別等についてご記入ください。後の製品開発の参考にさせていただきます。

４．障害情報をお送りいただく場合

最初の一行は「障害情報」とのみ書いてください。障害情報の場合は、マイクロソフト社内で再現テストを迅速に行なうため、わかる範囲でできるだけ詳細に記入してお送りください。また場合により、ご報告の障害情報について弊社より電子メールでお尋ねさせていただく場合があります。

【障害のタイトル】

  • 障害の内容について、簡潔に一文で記入してください。

【障害の内容と再現手順】

  • 障害の状況を自由形式で記入してください。
  • 障害発生の際に、もともと期待していた動作は何かを記入してください。
  • 障害発生頻度について記入してください (同じ手順で必ず発生、特定の状況で発生、不特定の状況でランダムに発生など)。
  • 障害再現手順は箇条書きにして、行なった操作を順を追って記入してください。

【ハードウェア仕様】

  • マシンのメーカー名と機種名 （もしあれば）
  • CPU (CPU 製品名、クロック数、CPU の数など)
  • メモリ容量
  • ハードディスクの容量及び各ドライブの容量
  • ディスプレイの解像度および色数
  • DVD ドライブの有無
  • ネットワーク機能の有無
  • ローカル プリンタの有無 (「有り」の場合、プリンタのメーカー名と機種名)

【ソフトウェア仕様】

  • OS の種別及び Service Pack のバージョン (例: Windows 2000 Professional Service Pack1 など)
  • OS インストール方法 (例: プリインストール、旧バージョンからのアップグレード、新規インストール)
  • 共存している主なソフトウェアの製品名 (例: Microsoft Office、ウィルス チェック ソフト、その他わかる範囲で)

５．ご注意

感想と障害情報は、一つの電子メールに何件でも入れていただいてけっこうです。その場合は、項目ごとに「感想」か「障害情報」かを明記してください。ハードウェア仕様、およびソフトウェア仕様は、障害情報の件数に関わらず、障害再現に利用したマシン台数分の情報をわかる範囲で記入してください。

弊社では、このベータの障害情報をお送りいただいた方に対して、対処結果を個別にお返ししておりません。また上記電子メールアドレスにご質問をお送りいただいた場合も、個別の回答はいたしかねます。あらかじめご了承ください。

バージョン情報

リリースの目的

このリリースは、Microsoft .NET Framework SDK のテクニカル プレビューを目的とし、評価用に開発されています。このリリースには、リリース前のコード、ドキュメント、および仕様が含まれています。このリリースでは、最終的な製品のパフォーマンスは実現されておらず、互換性についても保証されていません。

SDK のこのベータ バージョンでは、.NET Framework のセキュリティ アーキテクチャは完成されていますが、アーキテクチャの実装は完成されていません。この製品をインストールするすべてのコンピュータで、次の各項目に注意することを強くおすすめします。

  • インターネット上に存在するコードなど、開発元が信頼できないコードは実行しないようにしてください。
  • セキュリティ ポリシーを変更して、インターネット上に存在するコードなど、開発元が信頼できないコードに高レベルの権限を与えないでください。
  • Internet Explorer で Web ページを参照すると、コードがダウンロードされ、実行されることがあります。このため、信頼できるサイトにだけアクセスするようにしてください。インターネットに参照することで、不正なコードが実行されないようにするには、ActiveX コントロールの実行を無効にします。次のように設定してください。
    1. Internet Explorer で、[ツール] メニューの [インターネット オプション] をクリックします。
    2. [インターネット オプション] ダイアログ ボックスの [セキュリティ] タブをクリックします。
    3. [インターネット] アイコンをクリックします。
    4. つまみの設定を [高] に設定し、[OK] をクリックします。

リリースの出荷媒体

このリリースの .NET Framework SDK は、Microsoft Visual Studio.NET の Windows Component Update CD に収録されています。また、Web サイトからダウンロードして入手することもできます。

ローカライズ情報

.NET Framework SDK のベータ 1 リリースには、英語、ドイツ語、日本語の各バージョンが用意されています。.NET Framework SDK ベータ 1 では、その中核となるソフトウェアの一部がローカライズされています。ツール、サンプル、ドキュメントなど、その他の部分はローカライズされていません。

メモ: これら 3 つのバージョンでは、エラーメッセージなどのリソースが異なっています。上記の 3 ヶ国語バージョンのいずれも、中核となる機能や動作はほぼ同じです。このため、どのバージョンを選択しても、各国語対応アプリケーションを開発することができます。

セットアップ

インストール情報

.NET Framework SDK ベータ 1 は、実際の開発用または業務データが保存されているコンピュータにはインストールしないでください。

メモ: ASP.NET は、Windows 9x オぺレーティング システムではサポートされていません。これには、Windows Me、Windows 98 Second Edition、Windows 98 および Windows 95 が含まれます。

.NET Framework SDK ベータ 1 をインストールする前に、以下の手順に従ってください。

  • 「確認済みの問題」の「セットアップに関する問題」を参照します。
  • .NET Framework SDK の旧バージョンをすべてアンインストールします。SDK の PDC Tech Preview リリースを使用している場合は、SDK とともにインストールされる「PDC 以降ベータ 1 がリリースされるまでに変更された API」 を参照してください。
  • いずれのプラットフォームを使用する場合も、.NET Framework SDK ベータ 1 のセットアップを実行する前に、Microsoft Internet Explorer 5.5 と MDAC 2.6 がインストールされていることを確認します。

インストール前に必要なソフトウェアは、プラットフォームごとに次のようになります。

  • Windows 2000
    • IIS
      ASP.NET の機能を使用するには、.NET Framework SDK ベータ 1 をインストールする前に、Internet Information Server (IIS) 5.0 をインストールしておく必要があります。SDK のインストール時に IIS がコンピュータにインストールされていない場合、ASP.NET は正しく登録されず、ASP.NET アプリケーションは実行されません。SDK をインストールしてから IIS をインストールした場合は、regsvr32 を使用して xspisapi.dll ファイルを手動で登録する必要があります。

  • Windows NT 4.0
    • Service Pack 6a
    • Windows NT 4.0 Option Pack
      ASP.NET アプリケーションを実行するには、NET Framework SDK をインストールする前に Windows NT 4.0 Option Pack をインストールする必要があります。SDK のインストール時に IIS (Option Pack の一部) がコンピュータにインストールされていない場合、ASP.NET は正しく登録されず、ASP.NET アプリケーションは実行されません。SDK をインストールしてから IIS をインストールした場合は、regsvr32 を使用して xspisapi.dll ファイルを手動で登録する必要があります。

サンプルのインストール

サンプルは、以下の順序でインストールできます。サンプルのインストールに必要な各ファイルは、.NET Framework SDK を既定のフォルダにインストールした場合、システム ドライブの [Program Files\Microsoft.Net\Samples] ディレクトリに保存されます。

  1. サンプルに含まれる各ファイルを展開します。
    [Program Files\Microsoft.NET\Samples] ディレクトリの Sample.exe を実行します。
  2. 必要に応じて、MSDE をインストールします。
    サンプルの中には、SQL と ADO.NET を使って、SQL Server の使用方法を説明するものがあります。これらのサンプルを実行するには、SQL Server version 7.0 以降、または MSDE をインストールする必要があります。お使いのマシンに、いずれかのソフトウェアがインストールされていない場合は、MSDE をインストールします。MSDE をセットアップするには、[Program Files\Microsoft.NET\Samples\Setup] ディレクトリにある InstMSDE.exe を実行します。
    メモ： この MSDE は、英語バージョンです。この MSDE をインストールすると、システムの一部で、英語のメッセージやダイアログ ボックスが表示されます。
  3. サンプルのインストールを完了します。
    [Program Files\Microsoft.Net\    Samples\Setup] ディレクトリにある Finish.exe を実行してサンプルのインストールを完了します。この操作を行うと、[Setup] ディレクトリにあるデータベースがデータベース サーバーにインストールされ、IIS のバーチャル ディレクトリが作成されます。その後、NMAKE が別のウィンドウで実行され、サンプルがコンパイルされます。NMAKE が終了するまで、ほかの作業を行わないでください。

.NET Framework SDK を使用する前に

.NET Framework SDK をインストールすると、インストールされる SDK Start page には、.NET Framework SDK ベータ 1 を紹介するページです。このページには、プログラミング技術情報、サンプル コード、開発ツール、ユーティリティなど、Microsoft .NET Framework を使用してアプリケーションを開発するために必要なすべての情報へのリンクが用意されています。

セットアップに関する問題

  • インストール時に内部エラー 2359 が発生します
    問題:
    このエラーは、Windows ディレクトリまたは WWW ルート ディレクトリへの書き込みに必要な権限が、ローカルの SYSTEM アカウントにないために発生します。
    解決方法:
    [WINNT] ディレクトリおよび [wwwroot] ディレクトリのプロパティを表示します。[セキュリティ] タブで、ローカル SYSTEM アカウントに "フル コントロール" 権限が設定されていることを確認します。次に、セットアップを再実行します。

  • .NET Framework SDK をアンインストールすると、Visual Studio.NET 7.0 が動作しなくなります
    問題:
    .NET Framework SDK をインストールしてから Visual Studio.NET 7.0 をインストールし、その後 .NET Framework SDK をアンインストールすると、Visual Studio.NET 7.0 は正しく動作しなくなります。
    解決方法:
    Visual Studio.NET を継続して使用する場合は、.NET Framework SDK をアンインストールしないでください。Visual Studio.NET は、.NET Framework に依存しているため、.NET Framework がインストールされていない場合は正しく機能しません。

  • SDK をアンインストールしてもサンプルが削除されません
    問題:
    SDK をアンインストールしても、コンピュータからサンプルが削除されません。
    解決方法:
    サンプルを削除するには、以下の手順に従ってください。
    1. [コントロール パネル] の [アプリケーションの追加と削除] アプレットを使用して、Microsoft .NET Framework SDK サンプルをすべて削除します。
    2. [コントロール パネル] の [アプリケーションの追加と削除] アプレットを使用して、MSDE をアンインストールします。

  • アンインストール時に、セットアップ プログラムが World Wide Web 発行サービスを再起動しません
    問題:
    SDK がアンインストールされる前に、セットアップ プログラムは自動的に World Wide Web 発行サービスを停止します。しかし、アンインストールが完了しても、このサービスは再起動されません。手動でこのサービスを再起動する必要があります。

  • PDC Tech Preview 版から .NET Frameword SDK ベータ 1 へのアップグレード
    問題:
    .NET Framework SDK ベータ 1 リリースでは、バージョン管理の方法が変更されました。このため、PDC Tech Preview 版の mscoree.dll が System ディレクトリに保存されたままになることがあります。この場合、コンパイラが誤ったバージョン番号を報告したり、ASP.NET のページで "The specified procedure could not be found." というエラー メッセージが表示されたりする可能性があります。
    解決方法:
    これを回避するには、System ディレクトリに保存されている mscoree.dll のバージョンを確認してください。mscoree.dll のバージョンが「2000.14.1812.10」の場合は、このファイルを削除し、.NET Framework SDK ベータ 1を再インストールしてください。

  • サンプルをセットアップする際にインストールされる MSDE
    問題:
    サンプルをセットアップすると、英語版の MSDE がセットアップされます。
    解決方法:
    英語版の MSDE を使って、サンプルを使用してください。

ADO.NET に関する問題

  • SQLPooledConnection を使用すると Dllhost.exe が暴走します
    問題:
    System.Data.SQL.SQLPooledConnection を使用するとき、オブジェクト プール メカニズムに存在する同期エラーのために、Dllhost.exe が暴走する場合があります。オブジェクトの最初のインスタンスを作成中に、プールされるオブジェクトを複数のスレッドで作成しようとすると、この問題が生じます。
    解決方法:
    プールされるオブジェクトの最初のインスタンスを作成してから、ほかのスレッドを開始します。この問題は、Windows 2000 Service Pack 1 より後のサービス パックで対応される予定です。

  • ADODataSetCommand.FillDataSetSchema から予測以上の列が返されます
    問題:
    ADO マネージ プロバイダおよび Microsoft Oracle OLE DB Provider を使用してスキーマ情報を DataSet に代入する場合、SELECT ステートメントに指定されていない余分な列が DataSet に追加されることがあります。SELECT ステートメントに主キーまたはインデックス付き列が含まれていない場合は、これらの列が検索されて行の最後に追加されます。ただし、FillDataSet を続けて呼び出す場合、これらの余分な列のデータは DataSet に追加されません。
    解決方法:
    余分な列へのアクセスを避けるか、あるいは SELECT ステートメントに主キーまたはインデックス付き列を組み込みます。
  • LoadDataSet の呼び出しを含んだコードをデバッガ内でデバッグすると、System.Data.ADO.ADOException がスローされます
    問題:
    LoadDataSet の呼び出しを含んだコードをデバッガ内でデバッグすると、System.Data.ADO.ADOException がスローされます。
    解決方法:
    デバッグ中に回避する方法は現在ありません。ご了承ください。

ASP.NET に関する問題

  • ASP.NET から SQLPooledConnection オブジェクトを使用する場合に、問題が発生します
    問題:
    ASP.NET から SQLPooledConnection オブジェクトを使用するには、SQLPooledConnection クラスから派生するオブジェクトの定義を含むアセンブリが共有名を指定され、グローバル アセンブリ キャッシュに登録されている必要があります。詳細については、.NET Framework SDK ドキュメントの「Assembly Concepts」を参照してください。

  • コントロールが完全に初期化される前にマネージ コントロールのプロパティおよびメソッドへアクセスする場合に、問題が発生します
    問題:
    HTML ページの Load イベントから、マネージ Windows Forms クライアント コントロールのメソッドまたはプロパティにアクセスしようとすると、競合 (race condition) が発生する場合があります。ある状況では、ページの Load イベントが送信されても、マネージ コントロールが完全に初期化されないことがあります (特に、オブジェクトの OnCreateControl メソッドが呼び出されないことがあります)。その結果、Load イベントで実行されるスクリプト コードが、コントロールの一部のメソッドまたはプロパティにアクセスできない場合があります。できる限り、コントロールが完全に初期化されるまで待ってから、コントロールのメソッドまたはプロパティへアクセスしてください。これは、JScript の確認済みのバグであり、マネージされていないコントロールにも、同様に影響します。

  • アプリケーションの再起動によって、ASP.NET のメモリがリークします
    問題:
    このリリースでは、アプリケーションが再起動されるたびに ASP.NET のメモリがリークします。ASP.NET では、config.web ファイル、global.asax ファイル、またはアプリケーションの \bin ディレクトリ内のアセンブリが変更されるたびに、アプリケーションが再起動します。この結果、最小でも 2 MB のメモリ リークが発生します。また、大きなアプリケーションの場合は、リークがさらに大きくなる可能性があります。

    ASP.NET のプロセス循環機能では、使用可能な仮想メモリの 40% 以上がプロセスによって消費されていることが検出されると、ASP.NET ワーカー プロセス (xspwp.exe) が再起動されます。再起動によって、メモリ リークは自動的に処理されます。再起動を引き起こすメモリの消費量の割合は、ルート config.web ファイルの セクションにある memorylimit 属性の値を変更することで設定できます。ルート config.web ファイルは、通常、%windir%\Microsoft.Net\Frameworks\version ディレクトリにあります。 また、プロセスの再起動をより積極的に行いたい場合は、指定した時間間隔、または一定の要求が行われた段階で、強制的にプロセスを再起動できます。 プロセスを強制的にリセットするには、ASP.NET ワーカー プロセス (xspwp.exe) を終了するか、または IIS を再起動します。

  • Windows NT 4.0 でトランザクションがサポートされていません
    問題:

セキュリティに関する問題

  • セキュリティが完全に信頼できない場合、Windows Forms などのマネージ コードを実行すると問題が発生します
    問題:
    既定では、マネージ コード (Windows Forms アプリケーションなど) をネットワーク フォルダ経由で実行するような場合、イントラネット ゾーンのセキュリティ ポリシーに従ってコードが実行されます。この状況で、権限のない操作が行われた場合は、セキュリティ例外が発生します。このリリースでは、セキュリティが完全でない場合にマネージ コードが正常に実行されるようにするには、セキュリティ ポリシーを変更する必要があります。

    概要:
    通常、.NET Framework クラス ライブラリでは、保護されたリソースにアクセスする場合、適切なアクセス権を要求することで、セキュリティの最適な実装が行われます。これまで、Windows Forms クラスでは、正しいアクセス権が常に要求されるとは限らない問題がありました。今回のベータ 1 では、セキュリティを確保するために従来の保守的な手法が採用されています。セキュリティに関連する Windows Forms メソッドは、再編成され、分類されています。これまで適切に保護されていなかったメソッドは、セキュリティ システムによって自動的に生成される UnmanagedCode アクセス権が必要となり、保護されるようになりました。つまり、Windows Forms の呼び出し元は、ベータ 1 の設定より高いレベルのセキュリティを必要とします。ランタイムが広範にわたって配布され、ランタイムと .NET Framework クラス ライブラリの完全性を維持して配布されることが重要であるため、このように慎重な方法でセキュリティの実装が行なわれます。

  • セキュリティの基本:
    アプリケーションが起動されると、セキュリティ システムによって妥当性がチェックされます。次に、セキュリティ システムはアプリケーションの証拠をチェックし、使用できる最も強固なフォームが検出されます。この証拠を使用して、そのアプリケーションへのアクセス権がセキュリティ システムからグループに付与されます。通常の場合、アクセス権の付与に使用される最も一般的な証拠は、アプリケーションが起動されるゾーンとなります。各ゾーンには、アクセス権のグループが割り当てられています。アプリケーションが実行されるコンピュータの管理者は、実行可能プログラムに対し、常にできるだけ直接セキュリティの変更を行って、二次的な障害を最小限にする必要があります。

    アプリケーションに追加の権限を付与するには、次の 2 つの方法があります。まず 1 つ目は、アプリケーションの証拠を変更する方法です。この方法は、アプリケーションごとに適応されるため、ご使用をお勧めします。この方法は、ネットワーク共有するコンピュータからローカル コンピュータにアプリケーションを移動するか、または厳密な名前を使って署名するだけで実行できます。2 つ目は、アクセス権のグループを直接変更する方法です。この方法は、ほかのアプリケーションに影響を与える可能性があるため、お勧めしません。

  • 解決方法:
    以下の作業のいずれかを実行してください。

    1. アセンブリに厳密な名前を割り当て、アクセス権のグループを作成します。
      1. コードの署名に使用する厳密な名前キーを作成します。コマンド プロンプトで、次のように入力します。
            sn -k myapp.snk

      2. カスタム属性を使用してコードにキーを追加し、コードを再コンパイルします。たとえば、以下のようにします。
            using System.Runtime.CompilerServices;
            [assembly: AssemblyKeyFileAttribute("myapp.snk")

        メモ : コードに名前空間が含まれている場合、using ディレクティブおよびカスタム属性は名前空間宣言の上に設定する必要があります。

      3. 上記の厳密な名前を持つコードに設定された Everything アクセス権を付与するためのコード グループを追加します。コマンド プロンプトで、次のように入力します。
            caspol -m -ag 1 -strong -file "myapp.exe" -noname -noversion Everything

    2. 特定のアプリケーション ドメインに追加のアクセス権を付与するには、コマンド プロンプトで次のように入力します。

          caspol -m -ag 1.2 -url "file:///\\computername/share/*" Everything

    3. コントロール ホスティングだけを許可する場合は、追加のアクセス権を持つ、信頼できるゾーンにサイトを移動します。以下の手順に従ってください。

      1. 信頼できるゾーンにサイトを移動します。
        1. Internet Explorer で、サイトに移動します。
        2. [ツール] メニューの [インターネット オプション] をクリックし、[セキュリティ] をクリックします。
        3. [信頼済みサイト] をクリックします。
        4. [サイト] をクリックします。
        5. [次の Web サイトをゾーンに追加する] ボックスに入力して [追加] をクリックし、サイトを追加します。
        6. ブラウザの右下隅の表示をチェックして、そのサイトが [信頼済みサイト] かどうかを確認します。

      2. 信頼できるゾーンに追加のアクセス権を付与します。コマンド プロンプトで、次のように入力します。

            caspol -cg 1.5 Everything

    4. イントラネット ゾーンに Everything アクセス権を付与します。コマンド プロンプトで、次のように入力します。

          caspol -cg 1.2 Everything

    5. セキュリティを無効にします。

      この処理は、最終的な解決方法として行うもので、トラブルシューティングや問題を特定する場合だけに役立ちます。コマンド プロンプトで、次のように入力します。

    次の状況で、InvalidCastException が発生します。まず、トランザクションを Required に設定し、親オブジェクトの新しいインスタンスを作成します。次に、親オブジェクトは、ContextUtil.CreateObject で、複数の COM+ プール子オブジェクトを作成します。子オブジェクトはトランザクション Supported とマークされます。この子オブジェクトには、データベース テーブルに対して動作する Method1 メソッドとインターフェイスが実装されます。Method1 は最初に呼び出されたときに正しく動作します。次に、インターフェイスを Null に設定して解放します。その後、別のテーブルに行を挿入するため、インターフェイスが別の子オブジェクトから呼び出されると、System.InvalidCastException がスローされます。
    解決方法:
    System.Runtime.InteropServices.Marshal.ReleaseComObject を使用して、マネージ オブジェクトで追加の解放を行います。

  • COM Interop を使用して、正常な HRESULT を取得した場合に、問題が発生します
    問題:
    既定では、マネージ コンポーネントから HRESULT は検出できません。COM コンポーネントから、エラーの HRESULT が返された場合、マネージ クライアントには、該当する例外がスローされます。COM コンポーネントから正常な HRESULT が返された場合、マネージ クライアントには、正常に完了したメソッドの呼び出しだけが検出できます。今後のリリースでは、この既定の動作を変更し、正しい方法として、System.Runtime.InteropServices の PreserveSigAttribute が使用されるようになる予定です。ただし、このリリースでは、PreserveSigAttribute にこの機能はありません。

    たとえば、マネージされていない COM コンポーネントが次の形式であるとします。
        HRESULT MyMethod();

    既定のマネージ シグネチャは、以下のようになります。
        void MyMethod (); // 既定では、[PreserveSigAttribute(false)]

    PreserveSigAttribute が適用された形式は、以下のようになります。
        [PreserveSigAttribute(true)]
        int MyMethod ();

    今後のリリースでは、後者の例で、例外がスローされることはありません。HRESULT は、成功または失敗に関係なく、常に返されます。
    解決方法:
    このリリースでは、ComImportAttribute とのインターフェイスに注意して、マネージ コード (PreserveSigAttribute があるもの、またはないもの) に後者の例の形式を使用し、HRESULT をマネージ コードに公開してください。

        "Assembly"="MyAssembly, Ver=1.0.0.0, Loc="""

        "Assembly"="MyAssembly, Ver=1.0.0.0, Loc=\"\""

  • 遅延バインディングが行われる Interop シナリオでは、COM からのカスタム マーシャリングがサポートされていません
    問題:
    CLR では、COM クライアントで使用する、IDispatch インターフェイスの 2 つの実装方法が用意されています。既定の実装方法では、カスタム マーシャリングはサポートされていません。カスタム マーシャリングされた引数を含むメソッドを COM から (既定の IDispatch インプリメンテーションを使用して) 呼び出した場合、MarshalAsAttribute は無視され、カスタム マーシャラは呼び出されません。遅延バインディングによる COM Interop シナリオでカスタム マーシャリングを要求する場合は、2 番目の IDispatch インプリメンテーション (オートメーション互換インプリメンテーション) を使用する必要があります。この "互換インプリメンテーション" は、アセンブリ レベルの属性を使用して、以下のような C# のコードで指定できます。

        using System.Runtime.InteropServices;
        [assembly:IDispatchImpl(IDispatchImplType.CompatibleImpl)]

    この属性は、クラス レベルでも適用できます。カスタム マーシャリングまたは IDispatch 実装方法の詳細については、COM Interop のマニュアルを参照してください。

  • Type Library Exporter およびマーシャラは、COM Interop でのデリゲートの処理方法が異なります
    問題:
    Type Library Exporter とマーシャラでは、デリゲートに関する既定の動作が異なります。現在、エクスポータでは、既定で _Delegate インターフェイス ポインタとしてデリゲートがエクスポートされます。ただし、マーシャラでは実行時に、関数ポインタがマーシャリング デリゲートに既定値として設定されます。
    解決方法:
    デリゲートに関するこの既定の処理を COM Interop シナリオで使用するには、アンマネージ コードを入力して関数ポインタを使用する必要があります。ただし、これらの関数ポインタは、アンマネージ コードの _Delegate インターフェイス ポインタにキャストする必要があります。

    たとえば、C# で記述されたマネージ デリゲートは、以下の例のようになります。
        public delegate void MyDelegate(int);
        public void GetDelegate(out MyDelegate callback);

    上記の例は、C++ を使用して、COM から以下のようにアクセスされます。
        typedef void (CALLBACK *MyDelegate)(long);
        MyDelegate lpfnCallback;
        obj->GetDelegate((_Delegate**)&lpfnCallback);

    また、マネージ シグネチャでの明示的な MarshalAsAttribute(UnmanagedType.FunctionPtr) または MarshalAsAttribute(UnmanagedType.Interface) の宣言は、エクスポータとマーシャラの両方の既定の動作をオーバーライドします。したがって、開発者は、デリゲート動作のマーシャリングとエクスポートを厳密に制御できます。

    関数の評価を中止するには、デバッグするプロセスを別のコンソールに配置する必要があります。CorDbg コンソールで CTRL キーを押しながら Break キーを押して、関数の評価を中止します。このとき、プロセスが実行されている必要があります。プロセスが停止している場合は、CTRL キーを押しながら Break キーを押しても、関数の評価は中止されません。

  • CorDbg におけるスレッドの中断には、問題があります
    問題:
    CorDbg threads コマンドによって、実際に終了段階に入ったときにスレッドが実行中であることが示される場合があります。suspend コマンドを使用しても、このようなスレッドの終了は停止されません。すべてのスレッドが中断されている場合、プログラムがハングアップしたように見えることがあります。この場合は、CTRL キーを押しながら Break キーを押して、プロセスを中断します。実行を継続するには、一部のスレッドを再開して現在のスレッドを設定する必要があります。
  • caspol ツールの動作は、ドキュメントと異なります
    問題:
    ドキュメントの内容とは異なり、ローカライズされた caspol ツールは、既定では、マシンのポリシー レベルではなく、ユーザーのポリシー レベルの管理を行います。ユーザーがマシンのアドミニストレータ権限を持っていても、このツールはユーザーのポリシー レベルを管理します。
    解決方法:
    マシン全体のセキュリティ ポリシーを参照したり、変更したりする場合には、"-machine" のオプションを指定します。

  • 新規レポート Web サービスの表示に関して、問題が発生します
    問題:
    実行時にレポート Web サービスを使用または表示すると、以下のエラーが表示されることがあります。

    Exception generated: (0x014d6f30) <<System.Exception>
    _className=<null>
    _exceptionMethod=<null>
    _message=(0x014d2118) "Unable to generate temporary class (result=1).
    c:\winnt\temp\gen348.0.cs(287,14) : warning CS0618: 'System.Currency' is obsolete: 'Currency is going away in Beta2.Use Decimal class instead.'

    <compilation debugmode="true">
    <assemblies>

    <add assembly="SeagateSoftware.Shared"/>
    <add assembly="SeagateSoftware.Remote"/>
    </assemblies>
    </compilation>

その他の問題

  • コード ダウンロードに必要なソフトウェア
    問題:
    Web ブラウザを使用してコントロールをダウンロードする場合は、コンピュータに以下のものをインストールする必要があります。
    • Internet Explorer 5 以降 (Windows 2000 および Windwos NT 4.0 の場合) または Internet Explorer 5.01 以降 (Windows 98 の場合)
    • .NET Framework SDK ベータ 1
    このソフトウェアがインストールされていない場合、ブラウザがハングアップしたように見えたり、アプリケーションが予測できない動作をしたりすることがあります。

  • URI 文字列に 2 つ以上のコロン (:) があると、System.URI クラスで例外が発生します
    問題:
    2 つ以上のコロン (:) を含む文字列が渡されると、System.URI クラスのメソッドで例外がスローされます。たとえば、以下の文字列を新規 URI オブジェクトの作成に使用すると、例外が発生します。

    URI MyUri = new URI("http://www.microsoft.com/r/av1/http://www.microsoft.com"); //例外をスローします

    解決方法:
    現時点では、解決方法はありません。この問題は、ベータ 2 リリースで修正される予定です。

  • VS.NET では、Windows Millennium および 98 へのログインが必要です
    問題:
    ユーザーがユーザー名を入力しないでログイン ウィンドウのキャンセルを選択した場合、操作中に VS.NET が暴走することが確認されています。
    解決方法:
    これらのプラットフォームで VS.NET を使用する場合は、常にログインするようにしてください。
  • アクセラレータ キーの動作に問題があります
    問題:
    日本語環境では、アクセラレータ キー が誤って表示され、動作しません。たとえば、[OK] が [OK&] となり、Alt+O キーを押しても、該当部分にジャンプしません。
    解決方法:
    この問題は、ベータ 2 リリースで修復される予定です。ご了承ください。
